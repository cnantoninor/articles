# Epistemic Debt in LLM-Generated Code

## Overview

**Status:** Drafting
**Created:** 2026-01-25
**Last Updated:** 2026-01-26

An exploratory article examining the epistemic risks of integrating LLMs into software engineering workflows. The core thesis is that LLMs represent an epistemological rupture—shifting developers from a "construction" paradigm (where understanding derives from authorship) to a "curation" paradigm (where understanding must be actively established).

## Key Questions

1. How does LLM-assisted coding change the nature of developer understanding?
2. What is "epistemic debt" and how does it differ from technical debt?
3. What practices can help maintain epistemic warrant in an LLM-augmented workflow?
4. How would we measure epistemic debt accumulation?

## Central Concepts

- **Epistemic Debt**: Code that works but nobody understands
- **Solutioning Trap**: Jumping to implementation before defining the problem
- **Construction → Curation**: Paradigm shift in how developers relate to code
- **Epistemic Boundaries**: Points where meaning translates (Intent → Spec → Code → Tests)
